# [Project Name]

## Overview
One-line description. What problem it solves. Target audience.

## Identity
- Color palette (primary, secondary, accent) with hex values
- Typography (heading font, body font)
- Visual style / aesthetic (e.g. minimal flat, pixel art)
- Tone of voice

## Stack
List each chosen technology with a one-line "why".

## Architecture
- Pattern and one-line justification.
- Monorepo folder layout. Feature-scoped folders live inside
  a `features/` folder, named by domain (e.g.
  `features/lists/`, `features/submissions/`,
  `features/users/`). Cross-cutting concerns sit in their
  own shared folders (`db/`, `auth/`, `lib/`, `config/`).
  Types live in one place. Never duplicate them across apps.

## Deployment
This project targets a PaaS (Railway, Render, Fly.io,
Vercel, or similar). No self-managed servers or custom
infrastructure-as-code.

## Working principles for AI agents
- Confirm requirements before assuming. If ambiguous, ask
  before coding rather than guess.
- Explain non-obvious decisions when suggesting changes.
  Routine edits don't need narration.
- Surface uncertainty. Flag tradeoffs and unknowns
  explicitly instead of papering over them.
- Match existing patterns. Mirror recent files in the same
  folder rather than inventing new shapes.
- Clean up when you touch a file. Remove unused imports,
  vars, params, and dead code.
- Use pnpm only. Reject npm, yarn, bun, or anything else.
- Block packages whose latest version was published less
  than 7 days ago. Wait for the dust to settle before
  using brand-new releases. Enforce this by setting
  `minimumReleaseAge: 10080` (minutes) in
  `pnpm-workspace.yaml`, or `minimum-release-age=10080`
  in `.npmrc`.

## Environment & secrets
- All secrets in env vars. Never commit `.env`.
- Validate env at startup. Fail fast if anything is missing
  or malformed.
- Keep a thorough, well-maintained `.gitignore` from day
  one. Cover `.env`, build outputs, dependencies, caches,
  IDE/OS files, logs, and any credentials. Audit it before
  every push.

## Code standards
- TypeScript strict mode. No `any` without a written reason.
- Small, focused functions. Meaningful names.
- Reusable components in a shared folder. Write once, not
  per consumer.

## Database & queries  [include only if a DB is used]
- Schema in one place. Use an ORM (Drizzle, Prisma, etc.).
- Paginate every list endpoint. Default 25, max 100.
  Cursor-based for public feeds, offset for admin tables.
- Index every column used in WHERE / ORDER BY / JOIN.
  Composite indexes most-selective first.
- Select only the columns you need. Skip heavy text/JSON
  unless required.
- Avoid N+1. Use relation loaders or batch lookups.
- Parameterize all writes. Wrap multi-statement writes in
  transactions.
- Connection pooling on. Least-privilege DB user in prod.

## Auth & permissions  [include only if auth is used]
- Use Better Auth for all auth flows (sessions, password
  reset, OAuth, etc.).
- Cookies: HttpOnly, Secure, SameSite=Strict.
- RBAC enforced on the server, not just client-side.
- Never return password hashes, tokens, or internal IDs the
  client doesn't need.

## Input handling  [include only if the app accepts user input]
- Validate every input server-side (TypeBox, Zod, etc.).
  Client validation is UX only.
- Sanitize HTML before storing or rendering (XSS).
- File uploads: validate MIME type, size cap, and content.
  Store in object storage (R2, S3).
- Rate-limit submission endpoints aggressively.

## Hardening
- Security headers via helmet or framework equivalent. Set
  CSP explicitly for the sources the app actually uses.
- HTTPS-only in production. Redirect HTTP at the edge.
- Never log secrets, PII, tokens, session IDs, or full
  request/response bodies.
- Encrypt sensitive fields at rest when applicable.
- Keep dependencies current (Renovate / Dependabot).
- Never use `eval()` or dynamic code execution.
